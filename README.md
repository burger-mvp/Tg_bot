# Telegram-бот

Бот реализован на `aiogram 3.x`, использует PostgreSQL через `asyncpg` и готов к запуску локально или в Docker/Railway.

## Возможности

- регистрация: кнопка «Начать», выбор одного из семи языков интерфейса, затем передача собственного номера через кнопку контакта;
- роли `user`, `admin`, `super_admin` и `trusted_seller`;
- уведомление всех администраторов о первой завершенной регистрации пользователя;
- FSM-создание поста с 1–30 фото/видео (включая смешанные альбомы), выбором категории, расчетом AED → USD и обязательным текстом доставки;
- модерация постов обычных пользователей супер-администратором;
- PostgreSQL-очередь: один пост в 30 минут с 09:00 до 22:00 и один повтор через 7 дней;
- тестовый режим очереди: проверка каждую минуту и повтор через 3 минуты;
- поддержка локального Telegram Bot API для видео до 2000 МБ.
- APScheduler, который безопасно восстанавливает задачи повторов после перезапуска Railway.

## Первый запуск локально

1. Создайте и активируйте виртуальное окружение Python 3.11+.
2. Установите зависимости:
   ```bash
   pip install -r requirements.txt
   ```
3. Откройте `.env` и замените все шаблонные значения настоящими данными.
4. Для новой базы выполните SQL из `structure.sql` в PostgreSQL.
5. Для уже существующей базы выполните миграции `001_add_user_language.sql`–`004_add_user_export_fields.sql` по порядку.
6. Запустите бота:
   ```bash
   python bot.py
   ```

## Обновление существующей Railway-базы

Перед деплоем версии с языками выполните в SQL-консоли PostgreSQL Railway содержимое файла:

```text
migrations/001_add_user_language.sql
```

Миграция `001` добавляет `users.language_code`. Затем обязательно выполните:

```text
migrations/002_add_post_queue.sql
```

Вторая миграция создает таблицу `post_queue`, добавляет метку первой регистрации и не удаляет существующие данные. Миграции `003` и `004` добавляют магазины, доверенную роль, новые языки и поля экспорта. Все миграции безопасны для повторного запуска.

### Сброс тестовых данных перед релизом

Файл `migrations/005_reset_test_data.sql` **безвозвратно** удаляет все записи из `users` и `post_queue`, затем сбрасывает `shop_counter`. Выполняйте его вручную в SQL-консоли Railway только после резервного копирования, если тестовые данные действительно больше не нужны. Первый новый продавец после этого получит `Shop 1`.

## Роли

- `super_admin` — пользователь с `SUPER_ADMIN_ID`;
- `admin` — пользователь, чей ID есть в `ADMIN_IDS`;
- `user` — любой другой пользователь.

При первом `/start` бот предлагает выбрать язык. После нажатия выбор сохраняется в `users.language_code`, затем бот запрашивает собственный номер через Reply-кнопку Telegram и сохраняет его в `users.phone_number`. Главное меню отображается только после завершения обоих шагов. В этот момент всем ID из `ADMIN_IDS` и `SUPER_ADMIN_ID` отправляется уведомление о новом пользователе.

Все зарегистрированные роли используют **Создать пост**: загружают от 1 до 30 видео, выбирают товар и вводят цену/описание. Для ДВС применяется наценка 10%, для кузовного — 15%. Обычный пользователь отправляет заявку супер-администратору на модерацию; администратор и супер-администратор ставят её в очередь сразу.

Очередь проверяется в `:00` и `:30` строго с 09:00 до 22:00 включительно. Часовой пояс задается переменной `SCHEDULER_TIMEZONE` и по умолчанию равен `Europe/Moscow` (московское время). После первой успешной публикации пост автоматически повторяется через 7 дней и затем удаляется из `post_queue`.

### Тестирование очереди

Для временного ускорения очереди задайте переменную:

```env
TEST_MODE=True
```

В этом режиме бот проверяет очередь каждую минуту, публикует в любое время суток и повторяет пост через 3 минуты. После успешного повтора пост удаляется из очереди. Для штатной работы удалите переменную или задайте `TEST_MODE=False`, затем перезапустите сервис.

## Railway и Docker

Передайте значения из `.env` как переменные окружения сервиса Railway. Файл `.env` намеренно исключен из Docker-образа и Git.

Для публикации добавьте бота в целевой канал как администратора с правом **«Публикация сообщений»**. В `CHANNEL_ID` укажите числовой ID канала, обычно начинающийся с `-100`.

Дополнительно можно задать:

```env
SCHEDULER_TIMEZONE=Europe/Moscow
```

### Локальный Telegram Bot API для файлов до 2000 МБ

Стандартный Bot API ограничивает медиа в этом боте 20 МБ. Для больших видео нужен отдельный сервис Local Bot API. Исходный сервер поддерживается Telegram/TDLib; образ `aiogram/telegram-bot-api` — готовая неофициальная Docker-упаковка этого сервера.

1. В том же Railway-проекте создайте сервис из Docker-образа `aiogram/telegram-bot-api:latest`.
2. Добавьте этому сервису переменные:
   ```env
   TELEGRAM_API_ID=<api_id из https://my.telegram.org>
   TELEGRAM_API_HASH=<api_hash из https://my.telegram.org>
   TELEGRAM_LOCAL=1
   TELEGRAM_HTTP_PORT=8081
   TELEGRAM_HTTP_IP_ADDRESS=[::]
   ```
3. Подключите постоянный Railway Volume к `/var/lib/telegram-bot-api`. Публичный домен этому сервису не нужен.
4. В настройках сети Local Bot API скопируйте его private domain целиком (например, `telegram-bot-api.railway.internal`). В Variables сервиса бота добавьте:
   ```env
   TELEGRAM_API_SERVER_URL=http://<private-domain>:8081
   ```
   Используйте фактическое внутреннее имя Railway, а не пример из строки выше.
5. Перед первым включением переменной остановите текущий сервис бота и выполните один раз:
   ```bash
   curl -X POST "https://api.telegram.org/bot<TOKEN>/logOut"
   ```
   Затем включите `TELEGRAM_API_SERVER_URL` и перезапустите сервис бота. Одновременная работа одного токена с обычным и локальным Bot API может привести к потере обновлений.

Local Bot API в режиме `TELEGRAM_LOCAL=1` разрешает загрузку файлов до 2000 МБ. Лимит также зависит от доступного места на Volume и ресурсов Railway.
